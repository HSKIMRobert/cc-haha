# Claude Code Haha v0.4.11

这是 `v0.4.10` 之后的桌面体验与 Agent 工作流版本，重点新增互动桌面宠物、完整的桌面 Agent 管理、逐 Agent 模型与思考强度配置，并更新 Claude 官方模型阵容。同时继续修复任务状态、SubAgent 实时详情、附件恢复、Provider 创建、上下文估算和 Windows 桌面恢复等问题。

相比 `v0.4.10`，本版本让后台任务和 Agent 配置更容易观察、调整和复用：宠物可以在独立悬浮窗口中反映当前任务状态，Agent 可以按用户或项目范围管理，并为不同职责分别选择模型、effort 和工具。会话恢复后会继续保留附件内容，任务与子代理的运行状态也会更及时地同步到桌面端。

## Highlights

- **互动桌面宠物**：新增 Electron 原生透明悬浮窗口，内置搭搭、弧弧、补补、回回 4 个角色；宠物会根据工作中、等待处理、失败和空闲状态切换动作，并可展开活跃任务、点击返回对应会话。
- **本地自定义宠物**：支持导入透明 PNG / WebP 制作轻动画宠物，也支持导入 `1536×2288` 的 v2 逐帧动画图集；可调整尺寸、动画开关和任务面板显示，整个导入过程在本地完成，不调用当前聊天模型。
- **完整的桌面 Agent 管理**：设置页可以浏览内置、用户、项目、插件和策略等来源的 Agent 及覆盖关系，并创建、编辑或删除用户级与项目级 Agent。
- **逐 Agent 模型与 effort**：每个自定义 Agent 可以独立继承或指定 Fable、Opus、Sonnet、Haiku、自定义模型 ID，以及 `low` 到 `max` 的思考强度和可用工具；配置已接入 Agent 定义、运行、恢复和团队执行链路 (#1062)。
- **Claude 官方模型阵容更新**：桌面端与 CLI 新增并识别 Fable 5、Opus 4.8、Sonnet 5 和 Haiku 4.5，补齐显示名称、1M 上下文、推理强度和 Provider 环境映射。实际可用性仍取决于账号权限与 Provider 支持。
- **任务与 SubAgent 更可观察**：运行中的 SubAgent 可以立即打开详情并持续看到新记录；Task V2 的完成状态、并发更新和临时轮询失败也会更稳定地反映在桌面端 (#1075, #1077)。

## Fixes

- 修复部分 Provider 返回空或全零 usage 时，上下文占用估算消失的问题；现在会保留运行模型锚点并回退到本地 transcript 估算，不把估算值误作费用统计 (#1063)。
- 修复首轮消息携带图片或文件时，服务端回放可能让用户消息重复显示的问题；兼容 Windows 路径、UUID 前缀上传文件和内联图片 (#1069)。
- 修复拒绝工具权限后，正常中断被误报为执行失败的问题；当前轮会正确结束并回到可继续输入的状态 (#1070)。
- 修复任务已经完成但状态没有及时更新、旧请求覆盖新状态，以及临时轮询失败时任务列表闪空的问题 (#1075)。
- 修复运行中的 SubAgent 详情长期停留在“开始中”、必须等结束后才能查看执行过程的问题；运行记录和工具调用现在会持续刷新 (#1077)。
- 修复 Provider 预设接口暂不可用时“添加服务商”按钮失效的问题；常用预设改为随桌面端内置，并保留可编辑的 Custom 入口 (#1078)。
- 加强 Windows 白屏、Renderer 异常退出、代理连接重置、终端和 IM WebSocket 生命周期的恢复与错误隔离；主窗口会尝试有限恢复，并为无法恢复的启动故障保留可见错误信息 (#1064, #1065, #1071)。
- 修复桌面端完全退出或恢复会话后，文件、目录和 PDF 附件只剩路径、丢失原始上下文的问题；外部反馈分享仍会过滤附件记录和内容。
- 修复宠物任务卡在 Windows 上被原生窗口 shape 裁切、关闭按钮被透明命中区遮挡，以及重启后宠物未保持置顶的问题。
- 修复 Agent 文件已经保存后仍被可选热重载阻塞、运行中新增嵌套 Git 仓库时项目 Agent 可能写到外层仓库，以及旧刷新结果覆盖新项目状态的问题。
- 修复本地 HTTP 连接在 Windows 长时间空闲后被复用时，Session、Task、Scheduled 等接口可能同时超时的问题；同时补齐请求取消和任务轮询的资源清理。
- Kimi Code 预设升级为 K3：使用 Coding API、`k3` 模型、262K 上下文和 API Key 鉴权，并补齐 required thinking、effort 与 max effort 能力。

## Features and Improvements

- 桌面宠物支持位置拖动、点击互动、视线跟随、右键关闭和窗口位置恢复；任务面板默认收起，可用数量徽标展开最近的活跃会话。
- 自定义宠物导入增加本地化文件选择框，以及针对格式、尺寸、文件大小、重复 ID、损坏图片和存储失败的明确错误提示。
- Agent 列表展示来源、生效/覆盖状态、模型、思考强度、工具范围和系统提示词；内置、插件与策略来源保持只读。
- Agent 编辑器支持选择项目目录，并从当前运行环境读取真实可用工具，按类别搜索和勾选，同时保留 MCP、自定义工具和 `Bash(git:*)` 等规则输入。
- Agent 配置保存后会刷新当前会话组件；刷新失败不会回滚已经保存的定义，并提供可重试提示。
- 模型选择器会合并官方目录、当前 Provider 配置和运行时角色模型，保留第三方 Provider 的自定义模型映射。
- 会话状态和任务投影增加取消、乱序和临时失败保护，降低快速切换会话或多请求并发时的状态回退。
- Electron 构建补充 sidecar 启动、Renderer 恢复、宠物 IPC、打包输出空闲和跨平台产物契约检查。

## Notes

- GitHub Release 正文以 `release-notes/v0.4.11.md` 作为唯一来源。
- 桌面端发布版本号来源是 `desktop/package.json`；root `package.json` 继续保留本地开发用的 `999.0.0-local`。发布时必须确保版本号、Git tag `v0.4.11` 和本文件严格一致。
- `Release Desktop` workflow 会构建 macOS、Windows 和 Linux 的 x64 / ARM64 安装包，并在完整资产和 updater metadata 校验通过后公开 Release。
- 自动更新验证路径是正式 `v0.4.10 -> v0.4.11`。发布完成后需核对 `latest-mac.yml`、`latest.yml`、`latest-linux.yml` 和 `latest-linux-arm64.yml`。
- macOS 正式公开发布依赖签名和公证 secrets；Windows 未配置签名时仍可安装和自动更新，但可能显示 SmartScreen 提示。
- 桌面宠物是 Electron 桌面功能，不在 H5 中运行；单图导入提供本地轻动画，不是 AI 逐帧动画生成。
- Agent 的 effort 是否完整生效取决于所选模型和 Provider；不支持的组合可能被降低或忽略。

## 安装

在本页面底部 Assets 中按系统和 CPU 架构下载：

| 平台 | 文件 |
|------|------|
| macOS Apple Silicon（M 系列） | `Claude-Code-Haha-0.4.11-mac-arm64.dmg` |
| macOS Intel | `Claude-Code-Haha-0.4.11-mac-x64.dmg` |
| Windows x64 | `Claude-Code-Haha-0.4.11-win-x64.exe` |
| Windows ARM64 | `Claude-Code-Haha-0.4.11-win-arm64.exe` |
| Linux x64 | `Claude-Code-Haha-0.4.11-linux-x86_64.AppImage` 或 `Claude-Code-Haha-0.4.11-linux-amd64.deb` |
| Linux ARM64 | `Claude-Code-Haha-0.4.11-linux-arm64.AppImage` 或 `Claude-Code-Haha-0.4.11-linux-arm64.deb` |

### macOS

双击 DMG，把应用拖入“应用程序”。正式签名和公证产物应直接通过正常 Gatekeeper 流程。

### Windows

双击 `.exe` 安装。建议直接以当前用户运行，不要使用“以管理员身份运行”。覆盖升级前先完全退出应用；如果安装器报告相关进程仍在运行，请根据提示关闭对应进程后重试。未签名构建出现 SmartScreen 时可选择“更多信息”→“仍要运行”。

### Linux

AppImage：

```bash
chmod +x Claude-Code-Haha-0.4.11-linux-x86_64.AppImage
./Claude-Code-Haha-0.4.11-linux-x86_64.AppImage
```

deb：

```bash
sudo apt install ./Claude-Code-Haha-0.4.11-linux-amd64.deb
```

ARM64 机器请换用对应的 `linux-arm64` 文件。
